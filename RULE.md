## 二、ISSUES.md 维护规则

### 结构

```
✅ 已修复（按修复时间倒序排列）
一、设计层面（Design）
二、架构层面（Architecture）
三、实现层面（Implementation）
四、游戏逻辑层面（Game Logic）
其他（P7/P8 等历史遗留）
```

### 何时更新

- 发现 **bug**、性能问题或安全隐患时
- 代码与**设计意图不一致**或有改进空间时
- **修复完成**后，将条目从分类移至 ✅ 已修复

### 优先级

- **🔴 高** — 影响正确性或游戏体验
- **🟡 中** — 影响维护性
- **🟢 低** — 整洁或边缘情况

### 新增问题

**分类 → 标题 → 描述（表现+根因） → 标记优先级**

### 修复后

**移入 ✅ 已修复 → 重写（修复前/修复后） → 标注文件位置 → 标题加 ✅已修复 → 如有教训标注教训见 LESSONS.md L{N}**

### 其他状态

- **Won't Fix** — 标记理由
- **Deferred** — **必须附带可验证的触发条件**，不得"以后再说"

## 三、LESSONS.md 维护规则

**内容定位：** 可跨项目复用的**模式/原则/教训**，非代码行记录。**读者是 AI，不是人类**——每条教训的目的是让后续会话的 AI 在决策中被引用，防止重复犯错。

**写法原则：** 采用**因果式**（做了什么→导致了什么后果→以后怎么做），否定式开篇比肯定指令更有效，因为绑定了失败场景。

**禁止**写入具体文件路径、项目特有常量、设计决策记录（这些在 DESIGN.md）。

### 何时新增

- 修复中能学到**通用教训**时
- 教训可以**跨项目复用**时

### 写法框架

每个条目：**问题背景 → 错误做法（可选） → 正确做法 → 为什么更好**

### 交叉引用

- ISSUES 修复产生新教训时，在 ✅ 已修复条目末尾标注 **教训见 LESSONS.md L{N}**
- LESSONS 条目有对应 ISSUES 时，标注 **参见 ISSUES.md #{ID}**

## 四、GAME.md 维护规则

### 何时更新

- 新增/修改任何**游戏数值**时
- 新增/修改**怪物、物品、技能**时

### 如何更新

**替换表格**数值 / **~~删除标记~~** 已移除机制 / **代码块**写 Rust 公式表达式

## 五、DESIGN.md 维护规则

**内容定位：** 记录**设计意图**——为什么这样设计，而非怎么用或怎么实现。每个条目回答一个"为什么"问题。

### 何时更新

- 每次做出有意义的**架构或设计决策**时（选择方案、舍弃方案、权衡取舍）
- 代码行为与记录的设计意图**不一致**时（修正或说明为何偏离）
- **ISSUES 修复涉及设计变更**时同步更新

### 如何更新

- **追加新段落**，不删除旧记录
- 废弃的方案用 ~~已废弃~~ 标记并注明替代方案
- **关联 issue ID 或 commit hash** 以便追溯

### 禁止

- 不包含**具体游戏数值**（在 GAME.md）
- 不包含**操作步骤和命令**（在 PROTOCOLS.md）
- 不包含**代码实现教训**（在 LESSONS.md）

## 六、RULE.md 维护规则

**内容定位：** 操作规则总纲，定义其他文档的维护规范。不包含具体技术实现。

### 何时更新

- 发现文档体系**遗漏**（如缺少某文档的维护规则）
- 现有规则可**优化**（信息密度、记忆点、准确性）
- 流程**新增或变更**时

### 修改约束

**必须遵循：先提议 → 确认 → 再执行。** AI 不得自行改动。有改进建议直接向用户报告。
