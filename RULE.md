# RULE.md — 项目操作规则

本文档定义项目文档的维护规范。与具体技术实现无关。

---

## 一、文档体系

| 文档 | 内容 | 读者 |
|------|------|------|
| `README.md` | 项目概述、架构、构建方式 | 新开发者 |
| `GAME.md` | 数值设计文档——所有游戏数值和公式 | 游戏设计师 |
| `DESIGN.md` | 设计意图——为什么这样设计 | 架构评审者 |
| `ISSUES.md` | 问题追踪——已知问题与已修复项 | 所有开发者 |
| `LESSONS.md` | 从修复中抽象出的经验教训 | 所有开发者 |
| `RULE.md` | 本文——文档操作规范和流程 | 所有开发者 |

---

## 二、ISSUES.md 维护规则

### 结构

```
✅ 已修复（按修复时间倒序排列）
一、设计层面（Design）
二、架构层面（Architecture）
三、实现层面（Implementation）
四、游戏逻辑层面（Game Logic）
其他（P7/P8 等历史遗留）
```

### 优先级标记

- `🔴` 高 — 直接影响运行正确性或游戏体验
- `🟡` 中 — 影响代码质量和可维护性
- `🟢` 低 — 整洁或边缘情况

### 新增问题

1. 确定问题分类（设计/架构/实现/游戏逻辑）
2. 给出简短标题
3. 描述问题表现和根因
4. 标记优先级

### 修复后

1. 将条目从分类位置移到 `✅ 已修复` 区域
2. 重写描述：说明**修复前**什么问题、**修复后**什么逻辑、效果是什么
3. **不标注文件位置**（不说"位置：file.rs:line"）
4. 格式：纯逻辑描述，少量代码对比（如果需要）
5. 在标题末尾加 `✅已修复`

### 已关闭但无需改动项

标记 `Won't Fix` 并说明理由。

## 三、LESSONS.md 维护规则

### 内容定位

抽象出可迁移到其他项目的经验教训。不是具体代码行记录，而是**模式/原则/教训**。

### 何时新增

- 修复一个能从中学到通用教训的问题时
- 教训可以跨项目复用

### 结构

按主题分章节，教训编号 `L1, L2, ...`。

### 怎么写

```
### L{N} — 标题

一段或多段描述。包含：
- 问题背景
- 错误做法（可选）
- 正确做法
- 为什么更好

可以用代码块展示对比，但不是必须。
```

### 禁止

- 写入具体文件路径和行号
- 写入项目特有的常量值（除非作为示例）
- 写入设计决策记录（这些在 `DESIGN.md`）

---

## 四、GAME.md 维护规则

### 何时更新

- 新增/修改任何游戏数值时
- 新增/修改怪物、物品、技能时

### 如何更新

- 可直接替换表格中的数值
- 删除的机制在表格行中标记 `~~已移除~~`
- 公式用 Rust 表达式（代码块）

---

## 五、提交规范

### 前缀

| 前缀 | 用途 |
|------|------|
| `feat:` | 新功能 |
| `fix:` | Bug 修复 |
| `refactor:` | 重构（不改变外部行为） |
| `docs:` | 文档/注释变动 |
| `chore:` | 构建/工具/依赖变动 |

### 粒度

每个 commit 对应一个逻辑变更。修复 + 相关文档更新放在同一个 commit。

## 六、设计决策记录

以下内容不写在 ISSUES 或 LESSONS 中，仅在 DESIGN.md 或 GAME.md 中记录：

- 为什么选择某种架构
- 为什么舍弃某种方案
- 平衡性取舍的依据
- 已知的有意非标准行为（如"升级满血"）
