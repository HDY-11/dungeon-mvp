## 二、ISSUES.md 维护规则

### 结构

```
✅ 已修复（按修复时间倒序排列）
一、设计层面（Design）
二、架构层面（Architecture）
三、实现层面（Implementation）
四、游戏逻辑层面（Game Logic）
其他（P7/P8 等历史遗留）
```

### 何时更新

- 发现 **bug**、性能问题或安全隐患时
- 代码与**设计意图不一致**或有改进空间时
- **修复完成**后，将条目从分类移至 ✅ 已修复

### 优先级

- **🔴 高** — 影响正确性或游戏体验
- **🟡 中** — 影响维护性
- **🟢 低** — 整洁或边缘情况

### 新增问题

**分类 → 标题 → 描述（表现+根因） → 标记优先级**

### 修复后

**移入 ✅ 已修复 → 重写（修复前/修复后） → 标注文件位置 → 标题加 ✅已修复 → 如有教训标注教训见 LESSONS.md L{N}**

### 其他状态

- **Won't Fix** — 标记理由
- **Deferred** — **必须附带可验证的触发条件**，不得"以后再说"

## 三、LESSONS.md 维护规则

**内容定位：** 可跨项目复用的**模式/原则/教训**，非代码行记录。**读者是 AI，不是人类**——每条教训的目的是让后续会话的 AI 在决策中被引用，防止重复犯错。

**写法原则：** 采用**因果式**（做了什么→导致了什么后果→以后怎么做），否定式开篇比肯定指令更有效，因为绑定了失败场景。

**禁止**写入具体文件路径、项目特有常量、设计决策记录（这些在 DESIGN.md）。

### 何时新增

- 修复中能学到**通用教训**时
- 教训可以**跨项目复用**时

### 写法框架

每个条目：**问题背景 → 错误做法（可选） → 正确做法 → 为什么更好**

### 交叉引用

- ISSUES 修复产生新教训时，在 ✅ 已修复条目末尾标注 **教训见 LESSONS.md L{N}**
- LESSONS 条目有对应 ISSUES 时，标注 **参见 ISSUES.md #{ID}**

## 四、GAME.md 维护规则

### 核心原则

**每个数字必须标注生产方式**，禁止裸数值。读者必须能从数字旁标记得知"它为什么是这个数、能不能改、改了影响谁"。

### 三级标注体系

每个数值后跟一个 `[标注]`：

| 标注 | 含义 | 何时使用 | 可修改性 |
|------|------|---------|---------|
| `[⃞计算]` | 由数学模型推导得出，输入不变则输出不变 | 公式、派生参数、模型输出 | 改模型参数→自动更新 |
| `[⃞直觉]` | 凭经验或参考设定的初始值，需标注依据 | 首次设计、无模型支撑的常数 | 需写明"拍的值"的理由 |
| `[⃞试调]` | 经试玩验证后调整的值，需标注调整轨迹 | 修复体验问题后的修正值 | 标注 `旧值→问题→新值` |

### 结构要求

```
## Gm{N} {子系统名}

### 核心公式
```表达式``` [⃞计算/⃞直觉/⃞试调]

### 模型说明（如有）
推导 + 锚定目标 + 灵敏度

### 参数表
| 参数 | 值 | 来源 | 依赖 |
```

### 何时更新

- **新增子系统的数字** → 首次标记为 `[⃞直觉]`，写明依据
- **改现有数字** → 先标 `[⃞试调]` 记录轨迹，再稳定后决定是否模型化
- **模型推导出新值** → 替换旧值为 `[⃞计算]`，删除原标注备注

### 禁止

- ❌ 裸数值不加标注
- ❌ 误标：推导值标 `[⃞直觉]`、直觉值标 `[⃞计算]`（前者掩盖无模型、后者伪造可信度）

## 五、DESIGN.md 维护规则

**内容定位：** 记录**设计取舍**——每条回答"**过去否了什么、未来往哪走**"。

> 每个条目兼具两种角色：**罗盘**（指明未来方向）和**护卫**（记录被否的弯路）。过去的设计历史为未来的合理性提供论证，而非怎么用或怎么实现。

### 何时更新

- 每次做出有意义的**架构或设计决策**时（选择方案、舍弃方案、权衡取舍）
- 代码行为与记录的设计意图**不一致**时（修正或说明为何偏离）
- **ISSUES 修复涉及设计变更**时同步更新

### 如何更新

- **追加新段落**，不删除旧记录
- 废弃的方案用 ~~已废弃~~ 标记并注明替代方案
- **关联 issue ID 或 commit hash** 以便追溯

### 禁止

- 不包含**具体游戏数值**（在 GAME.md）
- 不包含**操作步骤和命令**（在 PROTOCOLS.md）
- 不包含**代码实现教训**（在 LESSONS.md）

## 六、RULE.md 维护规则

**内容定位：** 操作规则总纲，定义其他文档的维护规范。不包含具体技术实现。

### 何时更新

- 发现文档体系**遗漏**（如缺少某文档的维护规则）
- 现有规则可**优化**（信息密度、记忆点、准确性）
- 流程**新增或变更**时

### 修改约束

**必须遵循：先提议 → 确认 → 再执行。** AI 不得自行改动。有改进建议直接向用户报告。

### 对编辑谨慎

### 编辑前检查清单

调用编辑工具前按顺序检查：

**① 本次修改是否涉及 bug 修复/新增功能/UI 变更？** 是 → ISSUES 条目已在开放区？否 → **先记录 ISSUES，再修改代码**。是 → 继续。纯文档/重构不改变行为 → 跳过。

**② 修改目标包含 RULE.md？** 是 → 用户已明确同意本次修改？否 → 先征求同意。是 → 继续。

**③ 完成后：** ISSUES 移至 ✅ 已修复 → git commit → 如有教训标注教训编号。
