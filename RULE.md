# RULE.md — 项目操作规则

本文档定义本项目中的文档操作规则、工作流要求和知识点管理方式。

---

## 一、文档体系与操作规则

### GAME.md — 数值设计文档

**用途：** 记录所有游戏数值——行动耗时、战斗公式、属性公式、经验曲线、掉落表、怪物属性。

**何时更新：**
- 新增/修改任何游戏数值时
- 新增/修改怪物种类时
- 新增/修改物品时
- 新增/修改技能效果时

**如何更新：**
- 直接在对应表格中替换数值
- 删除的机制标记为"~~已移除~~"（保留历史记录）
- 数值公式用代码块 ``` 写 Rust 表达式

### ISSUES.md — 问题追踪

**用途：** 记录当前实现中与设计意图不一致或有改进空间的问题。

**结构：**
```
✅ 已修复 — 所有已关闭的问题（保持历史记录）
一、设计层面（Design）
二、架构层面（Architecture）
三、实现层面（Implementation）
四、游戏逻辑层面（Game Logic）
其他 — P7/P8 等历史遗留
```

**优先级标记：**
- 🔴 高 — 直接影响运行正确性或游戏体验
- 🟡 中 — 影响代码质量和可维护性
- 🟢 低 — 整洁或边缘问题

**何时更新：**
- 发现新问题 → 添加到对应章节
- 修复问题 → 从原位置移到 ✅ 已修复，说明从什么改成什么
- 已修复条目不标注位置（不写"位置：file.rs:line"），只描述逻辑变更

**如何写已修复条目：**
```
### Ix — 问题简述

修复前：XXX 导致 YYY 问题。
修复后：改为 ZZZ，效果是 WWW。
```

### LESSONS.md — 抽象教训

**用途：** 从已修复问题中抽象出的**通用教训**，适用于本项目及类似项目。

**结构：**
```
一、Rust 语言层面（L1-L5）
二、架构设计（L6-L10）
三、ECS 使用（L11-L14）
四、游戏逻辑（L15-L18）
五、物品/装备（L19-L20）
六、调试与测试（L21-L22）
七、事件帧模式（L23, 计划中）
八、地图与地形（L24-L25）
九、存档兼容性（L26）
```

**何时更新：**
- 修复一个能从中学到通用教训的问题时
- 教训应当抽象——不描述具体代码行，只描述模式/原则
- 每个教训有编号（L1, L2, ...）

**如何写：**
```
### L{N} — 标题

一段或多段描述教训。包含：
- 问题背景
- 错误做法（可选）
- 正确做法
- 为什么

尽量用代码块展示正确和错误的对比。
```

## 二、工作流规则

### 阶段 1：需求分析与发散搜索
1. 分析用户的高层需求
2. 搜索最佳实践、开源库、架构模式
3. 构思 2-3 种可行方案，列出优缺点
4. 保存到记忆系统，标签 `#需求分析`、`#方案调研`

### 阶段 2：方案确定
1. 向用户提问澄清，展示初步方案
2. 等待用户确认
3. 确定最终执行方案
4. 保存到记忆系统，标签 `#最终设计方案`

### 阶段 3：代码实现
1. 按确定方案编写代码
2. 每步 commit
3. 保存到记忆系统，标签 `#实现细节`

### 阶段 4-5：测试与修复
1. 编写测试用例
2. 运行测试
3. 失败则分析、修复、重跑（至少尝试 3 次）
4. 保存到记忆系统，标签 `#测试记录`、`#修复日志`

### 阶段 6：交付前审查
1. 检索 `#最终设计方案`
2. 逐项审查接口/数据结构/流程与设计一致
3. 记录审查结果，标签 `#交付审查`
4. 重大偏差则返回阶段 3

### 阶段 7：交付
1. 总结实现路径和决策执行情况
2. 告知用户完成

## 三、硬性约束

1. **每一步后使用记忆工具保存/更新记录**
2. **禁止在未确定方案前编写代码**
3. 遇到错误时**禁止停止**，必须尝试至少 3 次修复策略
4. 交付前**必须**完成设计-实现对照审查
5. 记忆标签保持一致：`#需求分析`、`#最终设计方案`、`#实现细节`、`#测试记录`、`#修复日志`、`#交付审查`

## 四、已知设计决策（非错误）

以下问题已评估为"设计选择"，不作为 ISSUES 记录：

| 主题 | 决策理由 |
|------|---------|

| 噪声+元胞生成怪物 | 替代房间中心刷新，自然地散布在地形上 |

## 五、关键设计模式

### 1. Page 枚举 + (&page, key) 二元组匹配

用于处理多页面 UI 的按键分派。比 if 嵌套更清晰、编译器保证全覆盖：

```rust
enum Page { List(InvPanel), Detail(DetailSource, usize) }
match (&page, key.code) {
    (Page::List(_), KeyCode::Char(ch)) if ch.is_ascii_lowercase() => { /* 热键 */ }
    (Page::Detail(DetailSource::LeftInv, idx), KeyCode::Char('e')) => { /* 装备 */ }
    (Page::Detail(_, _), _) => {} /* 其他键忽略 */
}
```

### 2. tap-tap 输入模式

```rust
// 第一下 → 预览（设 PlayerPreview.kind）
// 第二下（同键）→ 确认（enqueue_if_absent）
// 不同键 → 取消前一个预览，设新预览
```

### 3. Priority-based arbitration

```rust
// 多个决策 system 各自写意图
// arbitration_system 按 priority 降序合并
// 相同 priority 随机排序（GameRng）
Flee = 200 > Chase = 100 > Wander = 50 > Wait = 0
```

### 4. 事件驱动队列推进

```rust
next_event_distance() // 查最小 av_remaining
→ advance(dist)       // 同步推进所有条目
→ pop_ready()         // 取出 av≤0 的条目
→ check_condition()   // 保活检查
→ execute_entry()     // 执行
```

## 六、RNG 使用规范（统一 GameRng）

| 用途 | 随机源 | 访问方式 |
|------|--------|---------|
| 暴击判定 | GameRng | `world.resource_mut::<GameRng>().random_f32()` |
| 掉落掷骰 | GameRng | `LootTable::roll(&mut rng)` 传入 |
| 游荡方向 | GameRng | `world.resource_mut::<GameRng>().random_range(0, 8)` |
| 仲裁排序 | GameRng | `rng.random_range(0, 255)` |
| 地图生成 | 局部 RNG | 用 `map_seed` 创建 |

**禁止：** 在任何地方调用 `rand::random::<f32>()`（系统熵，不可复现）。
