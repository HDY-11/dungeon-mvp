# 设计意图记录

本文档记录各设计决策背后的**意图**——为什么这么设计，而非怎么用。
数值和游戏规则见 [GAME.md](GAME.md)；
模块和 API 结构见 [README.md](README.md)。

---

## 1. 四层 Crate 拆分

**预期与实际依赖链：**
```
core ← action ← world ← render
```

**意图：** 按**关注点变化速度**分层。游戏数据（core）变化最慢，渲染（render）变化最快。行动规则（action）独立于世界生命周期（world），世界生命周期独立于渲染。

| 层 | 变化原因 | 不拆分的影响 |
|---|---|---|
| core | 很少改动（数据类型、公式） | 动渲染代码时不小心改坏战斗公式 |
| action | 添加新行动/技能时需要改动 | — |
| world | 添加新地图特征/怪物行为时 | — |
| render | TUI库升级/换框架时需要改动 | — |

**为什么不是两个 crate（core + everything else）：** 因为行动执行逻辑（dungeon-action）没有渲染依赖，世界生命周期逻辑（dungeon-world）也没有渲染依赖。如果合在一起，一个 ui.rs 的改动会触发生命周期逻辑的重新编译。对开发迭代速度有实际影响。

## 2. 移除全局 World，改为参数传递

**历史：** 曾经有一个 `OnceLock<RwLock<World>>` 全局对象 + `world!()` 宏（提交 `41f37f0` 引入，`f43b502` 移除）。

**移除原因：** RwLock 的不可重入性导致两种死锁模式（`advance_action_queue` 持锁调用 `execute_*`；`render_ui` 持锁调用子函数）。第一次遇到时打了"分两阶段取锁"的补丁，但设计上不可靠：**任何一个新函数如果忘记遵守约定就会引入死锁。**

**选择参数传递的理由：** Rust 的借用检查器在编译期保证不会同时持有 `&mut` 引用——这比任何运行时锁策略都更强。函数签名 `fn foo(world: &World)` 和 `fn bar(world: &mut World)` 明确表达了读写意图，不需要文档约束。

**不选择 thread_local 的理由：** `thread_local!` 只适合线程生命周期和 World 一致的单线程场景。如果将来要分线程（AI 计算、异步 IO），thread_local 的 World 会无法访问。

## 3. 行动系统设计哲学

行动系统是项目中最核心的设计，其演化经历了多轮思考（见提交历史中的多次 revert 和 fix）。

### 3.1 设计前提：ECS 组件式行动授权

**前提：** 在 ECS 中，实体 = id + 组件集，系统 = 行为。行动系统也是这套范式中的一环。

**设计选择：** 行动能力由组件赋予。`CanMove`、`CanChase`、`CanFlee`、`CanWander`、`CanWait` 等组件表达了"这个实体具有这种行动的权利"。系统只需遍历并检查哪种行动的条件满足。

**为什么不是"脑链"（顺序执行的行为树）：** 脑链一次只执行一条链，顺序判断。组件式条件是并行的——系统可以批量检查所有同种条件，完全只读、完全可并行。而且组件式更易扩展：给实体加一个新组件就赋予了新行为，不需要修改决策流程。

**为什么不是单独的决策组件：** 有人提出用一个集中式决策组件来决定实体做什么。这实际上是把分散在各 Action 组件中的决策权又收回一个中心，降低了可扩展性和可组合性。如果想加一种新行为，需要同时改决策组件——而组件式只需加一个新组件和一个新 system。

**为什么不是状态机：** 状态机把"状态"和"行动"耦合在一起（"进入追击模式 → 执行 CanChase"），本质上和单独的决策组件没有区别。它也把决策权从组件抢到了状态机系统手中。

**设计结论：** 决策权归组件。组件携带条件，系统批量检查，仲裁解决冲突。

### 3.2 AV = 反应时 + 耗时

**反应时的概念来源：** 灵感来自网络延迟——"就像网络延迟那样，表示这次行动到下次行动的延迟"。反应时是实体的属性，由敏捷派生。它让高敏捷角色"更快出手"。

**为什么要加耗时：** 如果只有反应时，所有行动的耗时都由反应时决定——快角色所有行动都快，慢角色所有行动都慢。这就降低了策略性。不同的行动应该有不同耗时（移动比等待快，技能比攻击慢）。

**AV = 反应时 + 耗时：** 两者相加为单一值入队倒计时。没有独立的冷却系统。

**问题来源（冷却系统为何被移除）：** 旧设计有"冷却计时器"和"队列推进"两个独立的时间维度。两个系统需要保持同步（提交记录中有多次"冷却空转"fix），而且冷却值需要与行动耗时的量纲对齐。AV 合一后，冷却作为独立概念消失——需要冷却的效果（如技能频率限制）通过调整行动耗时来实现。

**事件驱动推进：** `next_event_distance()` 查询最小剩余 AV，所有条目同步推进该距离，然后批量执行。固定 tick 在事件密集时浪费计算、稀疏时空转。事件驱动只在真正有事件时推进，天然零空转。

### 3.3 无前摇、无后摇

**思考过程：** 前后摇是动作游戏概念——前摇是行动前的延迟，后摇是行动后的延迟。后摇用来制造决策点（对手抓 timing）。但在回合制 Roguelike 中，玩家很少有能力在"怪物行动后"才决定干什么——一般在怪物行动前就已经想好了。就算要看怪物做什么，也是看它做完之后怎么样，而不是"现在这个怪物处于后摇状态，我要攻击它"。

**设计结论：** 后摇完全没有用。前摇已经被反应时覆盖。复杂动作（如施法）应通过组合多个行动（"准备施法" + "施法" + "施法完成"）来实现，而不是前后摇。组合行动的语义更清晰、更好管理。

### 3.4 保活检查

**问题：** 一个行动入队时条件满足，但到它执行时条件可能已不满足（目标怪物已被其他人杀死、目标格已被占用、目标已离开视野）。

**设计：** 执行前调用 `check_condition()` 验证条件。条件不满足 → 移出队列，不执行。产生这个行动的实体损失了已投入的 AV（相当于"白等了"）。

**类比：** 网络通信中的 keep-alive——每次发送前检查连接是否还活着。

**为什么不提前检查：** 入队时已经检查过一次。但 ECS 世界中，其他实体的行动可能在队列推进过程中改变世界状态。只有执行前这一刻的检查是可靠的。

### 3.5 决策冲突：优先级仲裁

**问题：** chase/flee/wander 三个决策 system 可能同时对一个实体产生行动意图（或者 chase 和 wander 对同一个实体）。需要解决冲突。

**设计：** 优先级数值仲裁。每个行动组件自带 `priority`（数值越小越优先）：
- Chase = 100（玩家在视野内就追）
- Flee = 200（HP < 25% 逃跑优先于追击）
- Wander = 50（兜底，没有更高优先级的行动时才游荡）
- Wait = 0（最终兜底，什么都不做也要等一回合）

**优先级的局限与合理化：** 优先级是静态的——不能根据游戏状态自动调整。理论上用"状态机"或"动态优先级"更灵活。但实践中，当前的行为种类（chase/flee/wander/wait）的优先级关系是固定的：逃跑最重要，追击其次，游荡再次，等待最后。没有哪个场景需要"追击优先级高于逃跑"。所以**静态优先级在这个阶段是合理的简化**。未来如果需要动态优先级，可以通过组件扩展（如 `PriorityModifier` 组件临时调整某类行动的优先级）。

### 3.6 无预测轴 / 无锁定轴

**旧设计：** 行动轴上有"预测区"（可能发生的行动）和"锁定区"（确定要执行的行动），玩家可以在这两个区域之间做决策。

**移除原因：** 预测与组件式决策不兼容。组件式决策是"当前帧检查条件"——系统只能知道"此刻哪些行动的条件满足"，无法准确知道"下一帧条件是否仍满足"。预测需要模拟未来的世界状态，这在 ECS 中非常复杂（需要克隆 World 或维护回滚日志），且收益有限。

**设计结论：** 所有显示在行动轴上的条目都是已锁定、必然要执行的（除非保活检查失败）。玩家看到的行动轴就是当前队列的快照。

### 3.7 行动队列的本质：查询系统

**本质：** 行动队列的核心不是"谁先执行"的顺序器，而是一个**查询系统**——查询"哪个实体已准备好行动"。AV 只是条件的一种，而非全部。重点不在于谁先谁后，而在于：
1. 哪种行动被选择去做（决策）
2. 何时做（AV 倒计时）  
3. 行动耗时和反应时（AV 的组成部分）

当前实现只覆盖了 AV 倒计时这一维度。从设计空间来看，这是最基础的可运作子集。

## 4. 并行怪物决策（Schedule）

**设计前状态：** `run_monster_decision()` 串行遍历所有怪物，每个怪物检查条件→仲裁→入队。O(n) 且无法并行。

**选择 Schedule 的理由：** 三种怪物行为（追击、逃跑、游荡）的检查条件**没有数据竞争**：
- chase 只读 Player 位置 + 怪物 Viewshed
- flee 只读怪物 Stats（HP比率）
- wander 只读怪物是否已被 chase/flee 选中

适当的并行度是 3（三种行为）+ 1（仲裁）。太少（串行）浪费 CPU 能力，太多（为每个怪物一个 task）调度开销超过收益。

**意图缓冲区模式：** 三个决策 system 各自写入独立的 `ChaseIntents`/`FleeIntents`/`WanderIntents` 资源，仲裁 system 串行合并。这种**生产者-消费者**模式相比共享队列更安全（无并发写冲突），并且使数据流显式可追踪。

**为什么保留串行兼容入口：** 测试和调试时串行版本更容易单步追踪。`run_monster_decision()` 的存在说明设计者认为"可调试性"比"代码整洁"更重要——这通常是一个正确的工程取舍。

## 5. 物品系统 — Registry + ItemStack

**背景：** 背包格子里存完整的物品定义（名称、描述、属性、图标…）→ 每个格子占用大量内存，网络传输（未来可能）重量大，且无法实现"同一个物品有多个"而不复制。

**设计：** 物品定义存在 `assets/items.json` 中，运行时加载为 `ItemRegistry`（`OnceLock` 全局单例）。背包里只存 `(item_id, count)`。

**选择 OnceLock 而非 ECS Resource 的理由：** 物品注册表是**纯数据**——它没有任何 ECS 依赖，不需要访问 World。放到 Resource 里意味着所有读操作必须先拿 World 引用，增加了不必要的耦合。`OnceLock` 是 Rust 标准库提供的"一次性全局初始化"，非常适合这种"启动时加载、之后只读"的场景。

**为什么装备不占背包格子：** 设计决策是"装备是角色的属性，不是背包的内容"。把装备放在背包里需要"标记某些格子已装备"的额外状态（is_equipped flag 或装备槽索引），增加状态管理复杂度。装备直接持有 ItemStack 使装备和背包两个关注点正交。

## 6. ItemPickup 作为独立实体

**设计：** 地面上的物品是独立的 ECS 实体，带 `ItemPickup`（存 ItemStack）、`Position`、`Renderable` 三个组件。不是背包 Inventory 的一部分。

**意图：** 实体组件模式使物品可以像其他实体一样被 FOV 系统看到、被 VisibleMemory 记录、被碰撞图排除（`rebuild_occupancy` 中 `pickup.is_none()` 过滤了它们，所以不阻挡移动）。如果物品是 Inventory 的一个列表，FOV 渲染需要特殊处理"如何渲染一个不在 ECS 中的位置"。

**代价：** 拾取时需 despawn 实体、生成掉落需 spawn 实体——比"移动指针"开销大。但拾取/掉落频率低（每层几次），开销可忽略。

## 7. tap-tap 输入

**问题：** 方向键同时承担"移动"和"攻击"两个语义。如果敲一下方向键就直接执行，玩家无法区分"我想走过去"和"我想攻击敌人"——攻击是移动的一个副效应。

**设计：** 第一次方向键设预览状态（显示"移动(0,-1)"或"攻击"），第二次同键确认入队。不同键取消前一个预览。

**为什么不直接用鼠标/菜单：** 终端 Roguelike 的核心交互是键盘盲打。菜单选择破坏了"看着地图敲键"的沉浸感。

**预览状态为什么是 ECS Resource 而非函数局部状态：** 预览需要跨帧存在（第一次按键到第二次按键之间需要渲染更新显示预览文本）。函数局部变量在 `process_key` 返回后即消失。ECS Resource 是最自然的"跨帧持久化"方式。

## 8. 碰撞图（Occupancy Map）独立于 Map

**设计：** `Map` 存 Tile 地形（`Wall`/`Floor`），`OccupancyMap` 存每个格子被哪个实体占据。

**意图：** 两个关注点变化频率不同：
- Map 地形在楼层生成后基本不变
- OccupancyMap 在每次移动/攻击/死亡后都变化

合并到一个结构意味着每次更新都要复制地形数据。分离后 `rebuild_occupancy()` 只关心实体位置，不碰地形。

**为什么每次行动后重建而非增量更新：** 增量更新容易漏边界条件（实体死亡、下楼、传送——虽然目前没有）。全量重建 O(n) 对 40×20 地图（800 格）的开销小于 1μs，比维护增量正确性的心智负担低得多。

## 9. 视野记忆双结构（MapMemory + VisibleMemory）

**设计：** 两个独立的记忆结构：
- `MapMemory`：记录哪些格子曾经被看到过（boolean 数组），用于渲染已探索区域的灰色墙壁/地板
- `VisibleMemory`：记录最后看到的实体（glyph/color/位置），用于渲染视野外但已知的实体

**为什么不分在一起：** 两个结构的数据类型、更新频率、生命周期不同：
- MapMemory 只增不减（探索过的格子不会"遗忘"）
- VisibleMemory 需要清理已死亡实体（否则会显示幽灵实体）

如果合并，cleanup 逻辑需要区分"地图记忆（不清除）"和"实体记忆（需清除）"，增加复杂性。

## 10. 50ms 按键去重 + 16ms 轮询

**设计：** 输入线程以 16ms 间隔轮询（≈60fps），连续两次相同按键间隔小于 50ms 则丢弃后一次。

**意图：** 终端键盘的物理按键会触发重复的 key-repeat 事件（长按时）。如果不做去重：
- 方向键长按 → 连续触发预览/确认/预览/确认 → 玩家瞬间移动多格
- 50ms 的窗口允许正常双次敲击（tap-tap 确认），但过滤掉键盘重复

**16ms 不是任意选择的：** 它与 60fps 的输入采样对齐。如果轮询间隔太长（如 100ms），玩家按下键后最多需要等 100ms 才能反应——明显可感知的延迟。16ms 是人感知不到的单帧延迟下限。

## 11. 装备槽配对背包满的处理

**设计：** 装备一个物品时，如果背包满了，卸载的旧装备会丢失（无法放回背包），操作被拒绝并显示"背包已满"。

**逻辑路径：**
1. 从背包移除 ItemStack
2. 装备到槽位（可能换下旧装备）
3. 尝试把换下的装备加入背包
4. 如果加入失败（背包满），**放弃本次操作，恢复原状**（把新装备放回背包，旧装备放回槽位）

**意图：** 原子操作语义——要么全部成功，要么全部失败。如果允许"装备成功但旧装备消失"，就会永久丢失物品。这是玩家最痛恨的游戏体验。

**为什么不先卸载再装备：** 如果先卸载旧装备（背包空出一个位置）再装备新装备，玩家可以用"装备格"作为额外背包空间。这是一个合法的游戏机制，但增加了实现复杂度——需要保证卸载和装备之间的状态一致性。

## 12. 存档 bincode + 完整快照

**设计：** 使用 `bincode` 直接将整个游戏状态序列化为二进制文件。存档是一个完整的快照。

**为什么不是 JSON/YAML：** 游戏状态中包含 Entity 引用（`Entity` 类型），JSON 无法原生序列化 ECS 关系。bincode 直接作用于内存结构，序列化反序列化零映射开销。

**为什么是完整快照而非增量/日志：** 增量存档需要维护"操作日志"或"变更集"，读档时需要重放。对于单人 Roguelike（存档频率低、状态量小），完整快照更简单可靠。存档后的操作不需要保证可回放。

**为什么存档包含地图 tiles 的拷贝而非引用：** Map 结构包含大数组（40×20 Tile），而且在反序列化时需要重建完整的 `Map` 对象。如果引用注册表中的地图定义，注册表可能在后续版本变化导致旧存档不兼容。

## 13. 线程局部 RNG + GameRng 并存

**设计：** `dungeon-core/src/global.rs` 中有一个线程局部的 `RefCell<SmallRng>`，同时 ECS 资源中也有 `GameRng { rng: SmallRng }`。

**意图（现实而非理想）：** 线程局部 RNG 用于仲裁 system 中的随机选择（`arbitration_system` 调用 `dungeon_core::global::rand_u8()`）。这是因为仲裁 system 作为 ECS System 不能直接访问 `GameRng` 资源（资源访问在 System 签名中声明，而仲裁 system 没有声明对 `GameRng` 的依赖）。

**隐患：** 两个 RNG 使用不同的种子策略，会导致难以复现的随机行为。正确设计应该是仲裁 system 也依赖 `Res<GameRng>`。当前状态是重构未完成的遗留。
